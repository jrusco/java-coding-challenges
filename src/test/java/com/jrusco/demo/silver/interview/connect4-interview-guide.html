Connect4Array

7×6 board · Moves array input · Detect 4-in-a-row · Return winner
U
Phase 1 — Understand
Read & Constrain the Problem ⏱ ~3 min
Read the problem out loud. Ask every clarifying question. Don't touch the keyboard yet.
A 7×6 Connect4 board. You receive an array of moves where each value is a column number. Players alternate: even-index moves belong to Player 1, odd-index to Player 2. Return 1 if P1 wins, 2 if P2 wins, 0 if no winner. A win is 4 consecutive discs in a row, column, or diagonal.
🎙 narrate "OK so we have a classic Connect 4 board — 7 columns, 6 rows."
  • Board dimensions? 7 cols × 6 rows — Java matrix is int[6][7] (rows first)
  • Return values? 0 = no winner, 1 = P1, 2 = P2
  • Diagonals count? Yes — horizontal, vertical, both diagonals
  • Column overflow? Assume valid input
  • Check both players? Only the player who just placed — a win can only be triggered by the last disc
Notes
✋
Keep your hands off the keyboard during this entire phase. Pause visibly after reading. Every second here saves three minutes in the Implement phase. Asking clarifying questions — even when you know the answers — demonstrates systematic thinking.
Interviewer signal

"Are you listening carefully and constraining the problem before writing code? Candidates who ask no questions either already know the problem or are too nervous to slow down. Question-asking — even self-directed — signals a systematic engineer."

M
Phase 2 — Match
Identify the Pattern ⏱ ~1.5 min
Name the problem type. Choose your data structures and justify them before planning the algorithm.
2D Grid Simulation Directional Traversal Early Termination
  • int[][] board — 6×7 grid, 0 = empty, 1 = P1, 2 = P2
  • int[] heights — next free row per column → O(1) placement vs O(rows) scanning
🎙 narrate "This is a 2D grid simulation with directional traversal."
Notes
Interviewer signal

"Does the candidate have a mental library of patterns to reach for? Naming '2D grid + directional traversal' signals fluency. Independently proposing the heights array signals experience — it's not in the problem statement, it comes from reasoning about cost."

P
Phase 3 — Plan
Design Before You Code ⏱ ~3 min
Decompose the problem, state the key insight, estimate complexity upfront. Still no typing.
Two Sub-Problems
Sub-problem 1
Build the board from the moves array
Iterate through moves. Place each disc in the column at the height tracked by the heights array. Player determined by move index parity.
Sub-problem 2
Detect a win after each placement
After each disc is placed, check only from that cell. If it causes a win, return immediately — no need to scan the full board.
🎙 narrate "The naive approach scans the full board after all moves — O(rows × cols) per check. The better approach: check for a win right after each placement, and check only from the newly placed disc outward."
Complexity
Time Complexity
O(n)
One pass through n moves. O(1) win check per move (4 axes × ≤3 steps).
Space Complexity
O(1)
Board is fixed 6×7. Heights is fixed 7 entries. Both constant regardless of input.
Notes
🗺
Narrate your full plan before typing a single character. Say: "Here's how I'm going to approach this…" and walk through both sub-problems. This gives the interviewer a chance to redirect you before you invest 8 minutes coding in the wrong direction.
Interviewer signal

"Can the candidate decompose a problem and sequence the sub-problems correctly? Stating complexity before coding — not after — shows forward-thinking. The bidirectional check insight separates pattern-matchers from candidates who actually reason through the problem."

2
Implement connect4() — the entry point
connect4()
public int connect4(int[] moves) {
    int[][] board = new int[6][7];
    int[] heights = new int[7];

    for (int m = 0; m < moves.length; m++) {
        int col = moves[m];
        int row = heights[col]++;
        int player = m % 2 == 0 ? 1 : 2;
        board[row][col] = player;

        if (checkWin(board, player, row, col)) {
            return player;
        }
    }
    return 0;
}
heights[col]++ gives the row and pre-increments in one expression — next disc in this column lands one row higher.
3
Implement checkWin() — 4 axes from the placed disc
checkWin()
private boolean checkWin(int[][] board, int player, int row, int col) {
    return countAxis(board, player, row, col, 0,  1) >= 4   // horizontal
        || countAxis(board, player, row, col, 1,  0) >= 4   // vertical
        || countAxis(board, player, row, col, 1,  1) >= 4   // diagonal ↗
        || countAxis(board, player, row, col, 1, -1) >= 4;  // diagonal ↘
}
The (dr, dc) pair encodes direction — the same pair passed negated gives the opposite direction on the same axis.
4
Implement countAxis() — bidirectional disc count
countAxis()
private int countAxis(int[][] board, int player, int row, int col, int dr, int dc) {
    return 1
        + countInDir(board, player, row, col,  dr,  dc)
        + countInDir(board, player, row, col, -dr, -dc);
}
The 1 accounts for the placed disc itself; negating dr and dc is why direction vectors are cleaner than enumerating all 8 cases.
5
Implement countInDir() — walk one direction up to 3 steps
countInDir()
private int countInDir(int[][] board, int player, int row, int col, int dr, int dc) {
    int count = 0;
    for (int i = 1; i <= 3; i++) {
        int r = row + i * dr;
        int c = col + i * dc;
        if (r < 0 || r >= 6 || c < 0 || c >= 7 || board[r][c] != player) break;
        count++;
    }
    return count;
}
Only 3 steps needed — countAxis already contributes 1 for the placed cell, and 1 + 3 = 4.
Notes
🧱

Interviewer signal

"Is the code structured, readable, and introduced in the order the candidate planned? Short methods with single responsibilities indicate engineering maturity. Narrating every decision — even the obvious ones — shows the thought process, which is ultimately what's being evaluated."
